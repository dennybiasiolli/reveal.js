The Time Paradox
Building Timezone-Safe Python/Django Applications

- or -
How your code can be wrong
without changing a single line

Denny Biasiolli

Handling time in software seems simple, until it isn't. From daylight savings quirks to database inconsistencies and user-local conversions, timezones can quietly turn into one of your biggest sources of bugs.
In this talk, we'll unravel the paradox of time in Python and Django: how something so universal can be so tricky to get right. You'll learn how Django handles datetime objects, the difference between naive and aware datetimes, common pitfalls when working with ORM and serializers, and best practices for ensuring your applications stay consistent across users, systems, and regions.
By the end, you'll be equipped with practical techniques and mental models to make your Django apps truly timezone-safe, and your logs, reports, and users finally in sync.

ISO 8601 Principles

  • Values are ordered from the largest to the smallest unit of time (YYYY-MM-DDTHH:MM:SS)
  • Each component is zero-padded to a fixed length
  • For reduced precision, components can be omitted from the end (YYYY-MM, YYYY-MM-DD)

A bug with no code changes

A user in New York sets a daily reminder at 9:00 AM.

Works perfectly for months.

One morning in March, it arrives at 10:00 AM.

Nothing in your code changed.
No config. No deployment.

The only thing that changed was the time in New York.

DST — Daylight Saving Time. The clock moved forward, but the stored UTC value didn't adapt. This is the core difference between recording the past and scheduling the future. Hints: https://www.youtube.com/watch?v=rPHZwB9rLYM

Two problems that look the same

  1. Recording when something happened
    order created, log entry, error occurred
  2. Scheduling when something should happen
    reminder, report generation, recurring task

They look similar, but they need different solutions.

Solution 1: Record in UTC

UTC is the universal reference time — it never shifts for daylight saving.

  • Unambiguous, always the same reference point
  • Convertible to any local timezone
  • The right choice for anything that already happened

Solution 2: Store the intent

The user doesn't want 14:00 UTC.

They want 9:00 AM in their day — every day, regardless of UTC offset.

If you store 14:00 UTC at creation time and DST changes, that becomes 10:00 AM local.

Instead, store 09:00 + America/New_York
and recalculate UTC whenever you schedule.

The intent stays constant. The UTC output adapts.

UTC-5 ≠ America/New_York

UTC-5 (fixed offset) America/New_York (timezone)
The offset at one specific instant A set of rules that change over time
Always UTC-5, forever UTC-5 Nov–Mar, UTC-4 Mar–Nov
→ Use timezone identifiers, not fixed offsets
These rules live in the IANA Time Zone Database (TZDB), the standard used by every OS and language.

IANA Time Zone Database

The standard behind every OS and programming language.

It defines the rules for every timezone:
when DST starts and ends, what offsets apply, historical changes.

When you use "America/New_York", your runtime looks up the rules automatically.

This is why "America/New_York" stays correct for dates in 1970, today, or 2040.

DST creates two tricky edge cases

  1. Spring forward — the clock skips an hour.
    A point in time that never existed.
  2. Fall back — the clock repeats an hour.
    A point in time that happened twice.

Spring Forward: the phantom hour

In New York, DST starts on March 9, 2025 at 2:00 AM.

The clock jumps: 1:59:59 AM → 3:00:00 AM

2:30 AM on March 9, 2025 never happened in New York.

If a user schedules something at 2:30 AM that day:

  • Some libraries throw an exception
  • Others silently move to 3:30 AM

Know what yours does. Warn the user.

Fall Back: the duplicate hour

In New York, DST ends on November 2, 2025 at 2:00 AM.

The clock jumps back: 1:59:59 AM → 1:00:00 AM

1:30 AM on November 2, 2025 happens twice in New York.

Which one did the user mean? This is ambiguous.

  • Pick a default (first or second occurrence)
  • Or ask the user to clarify

Days aren't always 24 hours

  • March 9, 2025 in New York: 23 hours long
  • November 2, 2025 in New York: 25 hours long

This matters when scheduling the next run:

# skips/repeats the DST hour
next_run = now + timedelta(hours=24)

# respects the local calendar day
next_run = now + timedelta(days=1)

Timezone concepts: recap

  • Past events → store in UTC
  • Future scheduling → store local time + timezone identifier, recalculate UTC when needed
  • A fixed offset is a snapshot; a timezone identifier carries the rules
  • Test DST edge cases — if your code handles those, it handles everything

Timezone in Python

datetime object

  • naive, no timezone information
    using the system's local timezone
    to interpret the time
  • aware, with timezone information
    using the specified timezone
    to interpret the time

Example

from datetime import datetime
from zoneinfo import ZoneInfo # Python 3.9+

# 09:00 local timezone
datetime(2025, 10, 26, 9, 0)

# 09:00 in New York
datetime(2025, 10, 26, 9, 0,
         tzinfo=ZoneInfo("America/New_York"))

Current timezone

In Python it's tricky to get the system's local timezone in a cross-platform way.

You can use third-party libraries like tzlocal 

import tzlocal

tzlocal.get_localzone()

Timezone in JavaScript

Date object

Tracks time in UTC internally, but typically accepts input and produces output in the local time of the computer/browser it's running on.

It has very few facilities for working with time in other time zones.

Example

new Date() // current date and time in local timezone

new Date(2025, 9, 26, 9, 00) // local timezone
new Date("2025-10-26T08:00:00.000Z") // UTC
Note: months are 0-indexed (January is 0, December is 11)

Current timezone

Intl.DateTimeFormat().resolvedOptions().timeZone

Working with other timezones

You can use third-party libraries like luxon or Day.js to handle timezones more effectively. 

// import { DateTime } from "luxon";
const { DateTime } = require("luxon");

DateTime.now().setZone("America/New_York");
DateTime.fromObject(
  { year: 2025, month: 10, day: 26, hour: 9, minute: 0 },
  { zone: 'America/New_York' });
Note: in Luxon, months are 1-indexed (January is 1, December is 12)

Timezone in PostgreSQL

timestamptz column

You can handle timestamps with timezone with the
timestamp with time zone
(aka timestamptz) type.

timestamptz column

It doesn't store the timezone itself,
but it normalizes the timestamp to UTC upon storage
and converts it back to the client's timezone upon retrieval.

PostgreSQL example

Let's say we have people working remotely
from different timezones,
spread from Los Angeles to Sydney.

Everyone stores and retrieves records
using a timestamptz column. 

insert into test_timezone (value_dt, comment) values
('2025-01-01T00:00:00-05:00', 'start of Q1 2025 in NY'),
('2025-03-31T23:59:59.999999-04:00', 'end of Q1 2025 in NY'),
-- ...
-- you can even use the timezone name directly
-- '2025-01-01 00:00:00 America/New_York'

  • scripts/02-get-js-date-ranges.js
  • scripts/03-insert-db-values.sql
  • select * from test_timezone order by value_dt;

If we query the table, PostgreSQL will convert the `timestamptz` value to the local timezone of the database (Europe/Rome in my case). 

select * from test_timezone order by value_dt;
--            value_dt            |           comment
-- -------------------------------+------------------------
--  2025-01-01 06:00:00+01        | start of Q1 2025 in NY
--  2025-04-01 05:59:59.999999+02 | end of Q1 2025 in NY
-- ...

-- check the current timezone with
show timezone;
--   TimeZone
-- -------------
--  Europe/Rome

Where is the timezone of the database configured?

  1. Server level in the postgresql.conf file
    timezone = '<tz>'
  2. Connection string
    ?options=-c%20TimeZone=<tz> 
    psql "postgresql://user@host/database?options=-c%20TimeZone=America/New_York"
  3. Database level
    ALTER DATABASE <db_name> SET TIMEZONE TO <tz>
  4. User level
    ALTER ROLE <user_name> SET TIMEZONE TO <tz>

A Django app typically shares one database connection pool.
You can't set a different timezone per user at the connection level.

Spoiler alert!
You can set it per transaction — and that's exactly what Django does under the hood.

So how can we do this more easily?
Setting the timezone for the session or transaction!

  1. For a single session
    SET TIMEZONE TO <tz>
  2. For a single transaction
    SET LOCAL TIMEZONE TO <tz>

This way you can get results in your desired timezone without changing the timezone permanently. 

begin transaction;
SET LOCAL TIMEZONE TO 'America/New_York';
select to_char(value_dt, 'YYYY-"Q"Q') as quarter, *
from test_timezone
order by value_dt;
end;
--  quarter |           value_dt            |    comment
-- ---------+-------------------------------+-----------------
--  2024-Q4 | 2024-12-31 08:00:00-05        | start Q1 Sydney
--  2024-Q4 | 2024-12-31 10:00:00-05        | start Q1 Tokyo
--  2024-Q4 | 2024-12-31 18:00:00-05        | start Q1 Rome
--  2024-Q4 | 2024-12-31 19:00:00-05        | start Q1 London
--  2025-Q1 | 2025-01-01 00:00:00-05        | start Q1 NY
--  2025-Q1 | 2025-01-01 03:00:00-05        | start Q1 LA

PostgreSQL + Django

When USE_TZ = True, Django wraps each query in a transaction that sets SET LOCAL TIMEZONE to the currently active timezone.

This is what makes per-user timezone possible with a middleware approach.

Django

Django has built-in support for timezones.
You can enable it in your settings with
USE_TZ = True

Django will store all datetime values in UTC in the database and will automatically convert them to the desired timezone when retrieving them.

Django Timezone setting

You can set the default timezone for your Django application using the TIME_ZONE setting in your settings.py file.

By default, it's set to 'UTC', but you can change it to any valid timezone string.

Django models

You can use the DateTimeField in your models to store datetime values.

When USE_TZ is enabled, Django will automatically handle the conversion to and from UTC for you.

Django hands-on

Let's create a simple Django model
to test timestamps and timezones. 

# models.py
from django.db import models

class TimezoneTest(models.Model):
    value_str = models.CharField(max_length=50)
    value_dt = models.DateTimeField()
    comment = models.CharField(max_length=100)

We can insert the same values we used in the PostgreSQL demo. 

from timezone_test_app.models import TimezoneTest

TimezoneTest(
    value_str="2025-01-01T00:00:00-05:00",
    value_dt="2025-01-01T00:00:00-05:00",
    comment="start of Q1 2025 in NY",
)
# ...

And we can query the records as well, to get all records related to Q1 2025 in New York. 

from timezone_test_app.models import TimezoneTest

TimezoneTest.objects.filter(
    value_dt__range=[
        "2025-01-01T00:00:00-05:00",
        "2025-03-31T23:59:59.999999-04:00"
    ]
).order_by("value_dt")

    or we will get a warning about naive datetimes.
  • 😭 We can't filter or group easily based on a specific timezone.

Django Middleware solution

You can create a custom middleware to set the timezone based on the user's preference or location. 

import zoneinfo
from django.utils import timezone

# read the timezone name from a custom header (API requests)
tzname = request.headers.get("X-Timezone")
if not tzname:
    # fallback to cookie (web requests)
    tzname = request.COOKIES.get("django_timezone")

if tzname:
    timezone.activate(zoneinfo.ZoneInfo(tzname))

In the frontend (JavaScript), you can set the timezone in a cookie or a custom header when making requests. 

// get the user's timezone
const tz = Intl.DateTimeFormat().resolvedOptions().timeZone;
// set timezone in a cookie
document.cookie = "django_timezone=" + tz;
// set timezone in a custom header (for API requests)
fetch("/api/some-endpoint", {
  headers: {
    "X-Timezone": tz,
  },
});

This way, Django will automatically convert the timestamps to the user's timezone when retrieving records.

Try navigating/filtering the admin interface while changing your browser's timezone settings. 

document.cookie = "django_timezone=Europe/Rome";

Try API filtering while changing the timezone header. 

curl -X GET "/api/timezone-tests/\
?value_dt__gte=2025-01-01\
&value_dt__lt=2025-04-01\
&ordering=value_dt" \
    -H "X-Timezone: Europe/Rome"

curl -X GET "/api/timezone-tests/group_by_year_and_quarter/" \

Three rules to live by

1. Store in UTC, display in local time.

A timestamp is a moment in time, not a timezone-aware value.
Convert to local only at the last moment, for human eyes.

2. For scheduling, store the intent.

Save local time + timezone identifier.
Recalculate UTC when the job runs, not when it was created.

3. Use timezone identifiers, not fixed offsets.

America/New_York knows the rules.
UTC-5 is just a number.

Bonus: test your DST edge cases.

Spring forward (the phantom hour) and fall back (the duplicate hour)
are where timezone bugs hide. Write tests for both.

If your code handles DST transitions, it handles everything.
