Handling time in software seems simple, until it isn't. From daylight savings quirks to database inconsistencies and user-local conversions, timezones can quietly turn into one of your biggest sources of bugs.
In this talk, we'll unravel the paradox of time in Python and Django: how something so universal can be so tricky to get right. You'll learn how Django handles datetime objects, the difference between naive and aware datetimes, common pitfalls when working with ORM and serializers, and best practices for ensuring your applications stay consistent across users, systems, and regions.
By the end, you'll be equipped with practical techniques and mental models to make your Django apps truly timezone-safe, and your logs, reports, and users finally in sync.

A bug with no code changes

A user in New York sets a daily reminder at 9:00 AM

Works perfectly for months

One morning in March, it arrives at 10:00 AM

Nothing in your code changed
No config. No deployment

What happened?

DST — Daylight Saving Time. The clock moved forward, but the stored UTC value didn't adapt. This is the core difference between recording the past and scheduling the future. Hints: https://www.youtube.com/watch?v=rPHZwB9rLYM

Wait, Daylight Saving Time!
The US and Europe change clocks on different weekends
so if you live in Europe and have users in the US,
it won't be your first guess that DST is the culprit

Two problems that look the same

  1. Recording when something happened
    order created, log entry, error occurred
  2. Scheduling when something should happen
    reminder, report generation, recurring task

They look similar, but they need different solutions

Recording when something happened:
Record in UTC

UTC is the universal reference time,
it never shifts for daylight saving

  • Unambiguous, always the same reference point
  • Convertible to any local timezone
  • The right choice for anything that already happened

UTC is the boring reliable friend who never cancels plans
Appreciate UTC

Scheduling when something should happen:
Store the intent

Users don't want 14:00 UTC

They want 9:00 AM in their day,
every day, regardless of UTC offset

If you store 14:00 UTC at creation time and DST changes, that becomes 10:00 AM local

Instead, store 09:00 America/New_York
and recalculate UTC whenever you schedule

The intent stays constant
The UTC output adapts

UTC-5 ≠ America/New_York

UTC-5 (fixed offset) America/New_York (timezone)
The offset at one specific instant A set of rules that change over time
Always UTC-5, forever UTC-5 Nov-Mar, UTC-4 Mar-Nov
→ Use timezone identifiers, not fixed offsets
These rules live in the IANA Time Zone Database (TZDB), the standard used by every OS and language.

IANA Time Zone Database

The standard behind every OS
and programming language

Defines rules for every timezone:
DST start/end, offsets, historical changes

In 2011, Samoa skipped an entire day (Dec 30th) by moving across the date line.
Someone had to update this database. Imagine the commit message

DST creates two tricky edge cases

(DST was invented by someone who hates programmers)

  1. Spring forward — the clock skips an hour
  2. Fall back — the clock repeats an hour

1. Spring Forward: the phantom hour

In New York, DST starts on March 9, 2025 at 2:00 AM

The clock jumps: 1:59:59 AM → 3:00:00 AM

2:30 AM on March 9, 2025
never happened in New York

1. Spring Forward: the phantom hour

If a user schedules something at 2:30 AM that day:

  • Some libraries throw an exception
  • Others silently move to 3:30 AM

Know what yours does and warn the user

2. Fall Back: the duplicate hour

In New York, DST ends on Nov 2, 2025 at 2:00 AM

The clock jumps back: 1:59:59 AM → 1:00:00 AM

1:30 AM on Nov 2, 2025 happens twice in New York

2. Fall Back: the duplicate hour

Which one did the user mean? This is ambiguous

  • Pick a default (first or second occurrence) If it's your alarm clock, pick the second one. You deserve the extra sleep
  • Or ask the user to clarify

Days aren't always 24 hours

  • March 9, 2025 in New York: 23 hours long
  • November 2, 2025 in New York: 25 hours long

This matters when scheduling the next run:

# skips/repeats the DST hour
next_run = now + timedelta(hours=24)

# respects the local calendar day
next_run = now + timedelta(days=1)

Timezone concepts: recap

  • Past events → store in UTC
  • Future scheduling → store the intent
    local time + timezone, calculate UTC when needed
  • Test DST edge cases
  • Offset is a snapshot
  • Timezone identifier carries the rules

Timezone in Python

datetime object

  • naive, no timezone information
    using the system's local timezone
    to interpret the time
  • aware, with timezone information
    using the specified timezone
    to interpret the time

A naive datetime assumes everyone works in the same timezone,
like people who think it's always sunny everywhere
because it's sunny outside their window

Example

from datetime import datetime
from zoneinfo import ZoneInfo # Python 3.9+

# 09:00 local timezone
datetime(2025, 10, 26, 9, 0)

# 09:00 in New York
datetime(2025, 10, 26, 9, 0,
         tzinfo=ZoneInfo("America/New_York"))

Timezone in JavaScript

Date object

Tracks time in UTC internally, but typically accepts input and produces output in the local time of the computer/browser it's running on

It has very few facilities for working with time in other time zones

"Very few facilities" is the polite way to say JavaScript's Date is a disaster.
The Temporal API is coming. Eventually. Probably. Don't hold your breath

Example

new Date() // current date and time in local timezone

new Date(2025, 9, 26, 9, 00) // local timezone
new Date("2025-10-26T08:00:00.000Z") // UTC

// how to get the current timezone of the browser?
Intl.DateTimeFormat().resolvedOptions().timeZone
Note: months are 0-indexed (January is 0, December is 11)

Timezone in PostgreSQL

timestamptz column

Short name for timestamp with time zone

Doesn't store the timezone itself

Normalizes the timestamp to UTC upon storage

Converts it back to the desired timezone upon retrieval

Configuring the timezone in PostgreSQL

  1. Server level in the postgresql.conf file
    timezone = '<tz>'
  2. Connection string
    ?options=-c%20TimeZone=<tz> 
    psql "postgresql://user@host/database?options=-c%20TimeZone=America/New_York"
  3. Database level
    ALTER DATABASE <db_name> SET TIMEZONE TO <tz>
  4. User level
    ALTER ROLE <user_name> SET TIMEZONE TO <tz>

Configuring the timezone in PostgreSQL

  1. For a single session
    SET TIMEZONE TO <tz>
  2. For a single transaction
    SET LOCAL TIMEZONE TO <tz>

Or it defaults to the system's local timezone

Django

Django has built-in support for timezones

You can enable it in your settings with

USE_TZ = True

All DateTime fields will be stored in UTC in the database and will be converted to the active timezone when retrieved

Django Timezone setting

You can set the default timezone for your Django application using the TIME_ZONE setting 

TIME_ZONE = "UTC"

By default, it's set to UTC, but you can change it to any valid timezone string

PostgreSQL + Django

When USE_TZ is enabled
Django wraps each query in a transaction that sets
SET LOCAL TIMEZONE
to the currently active timezone

The Problem in all of this

All users will see timestamps in the same timezone set in TIME_ZONE, regardless of their location

😞

Fiscal reports, calendar periods

This becomes a problem when you need to generate reports based on fiscal periods (e.g., quarters) that depend on the user's timezone

Fiscal reports, calendar periods

from django.db.models import Sum
from django.db.models.functions import TruncQuarter

# extract sum of all 2025 invoices, grouped by quarter
qs = (
    Invoice.objects.filter(value_dt__year=2025)
    .annotate(quarter=TruncQuarter("value_dt"))
    .values("quarter")
    .annotate(total=Sum("amount"))
    .order_by("quarter")
)
for record in qs:
    year = record["quarter"].year
    quarter = (record["quarter"].month - 1) // 3 + 1
    total = record["total"]

Solution #1

Timezone as a parameter for the fiscal report

  • 👍 Fiscal report in the desired timezone
  • 👎 Users have to specify the timezone every time they generate a report
  • 👎 It works only with the specific report request

Solution #2

Read timezone from user's browser (JS needed)
and set/send a cookie

  • 👍 Use the user's device timezone
  • 👍 No need to specify the timezone every time
  • 👎 Not ideal for API clients
  • 👎 Not ideal for third-party integrations
  • 👎 What if users need a different timezone?

Solution #3

Read timezone from a custom header
(e.g., X-Timezone)

  • 👍 Perfect for API clients
  • 👎 Not ideal for web users
  • 👎 Not ideal for Django admin

Solution #4

Timezone in profile settings

  • 👍 All timestamps in user's timezone
  • 👍 No need to specify the timezone every time
  • 👍 Can run reports for different timezones
  • 👎 Have to implement user settings for timezone

Django Middleware solution

Pick one, or combine them!

import zoneinfo
from django.utils import timezone
# read the timezone from user settings (if authenticated)
if request.user.is_authenticated:
	tzname = request.user.profile.timezone
# read the timezone name from a custom header (API requests)
tzname = request.headers.get("X-Timezone")
# read the timezone name from a cookie (web requests)
tzname = request.COOKIES.get("django_timezone")
# finally, set the timezone for the current request
if tzname:
    timezone.activate(zoneinfo.ZoneInfo(tzname))

In the frontend (JavaScript), you can set the timezone in a cookie or a custom header when making requests 

// get the user's timezone
const tz = Intl.DateTimeFormat().resolvedOptions().timeZone;
// set timezone in a cookie
document.cookie = "django_timezone=" + tz;
// set timezone in a custom header (for API requests)
fetch("/api/some-endpoint", {
  headers: {
    "X-Timezone": tz,
  },
});

Django will automatically convert the timestamps to the selected timezone when retrieving records

Three rules to live by

1. Use timezone-aware datetimes

Don't use naive datetimes, or you'll have to do timezone math manually

And timezone math is not real math!

2. Store in UTC, display in local time

UTC is the reliable, unambiguous way
to store timestamps

Local time is what users expect to see
when they look at dates and times

3. Ask for the timezone for sensitive operations
(like fiscal reports)

Don't rely on the server's timezone
or a fixed timezone for all users

Remember: timezone math is not real math!
